Lecture 12 · Part 2 · Exam Prep

SQLi — Exam Practice Questions

Scenario-based questions with model answers. Click any question to reveal the answer.

Quick Notes

  • Root cause:
    • Inline SQL = user input concatenated directly into SQL string → database cannot separate code from data
  • WHERE clause:
    • Filters rows by condition → if always true → every row returned
    • WHERE 1=1 or WHERE 'a'='a' = tautology attack → dumps entire table
  • Three SQL comment characters — attacker's eraser:
    • # → ignores everything after
    • -- → ignores everything after
    • /* */ → block comment
  • Three attack levels:
    • Bypass auth → admin'# → comments out password check → login without password
    • Dump data → ' OR 1=1# → tautology → returns every row
    • Modify/destroy → '; UPDATE...;# or '; DROP TABLE...;# → semicolon starts second statement
  • The only real fix:
    • Parameterized queries → query structure sent first with ? placeholders → data sent separately → database treats input as data only, never as code
    • Alice and Bob: "Only using parameterized queries reliably mitigates this vulnerability."
  • Second layer:
    • Least privilege → web app DB account is read-only → even if injection succeeds, cannot UPDATE or DROP
  • Three-layer architecture:
    • Browser → web server (concatenates SQL) → database
    • SQL injection uses web server as a proxy to reach the database the user cannot access directly
The exam trap:
"Input validation is enough" → Wrong → parameterized queries is the only real fix.
SQL injection = server-side in database ≠ XSS = client-side in browser.
Question Set 1

Login Bypass Scenario

Scenario

A developer writes this PHP login code:

$sql = "SELECT id, name, salary FROM credential WHERE name='$input_name' AND password='$input_pwd'";

A tester types admin'# into username and leaves password blank.

Q(a) 3 pts Write the resulting SQL query and explain why login succeeds without a password. ›
Model Answer
Resulting SQL query: SELECT id, name, salary FROM credential WHERE name='admin'#' AND password='' The single quote ' closes the name string literal. The hash # comments out the entire password check. The database evaluates only the WHERE name='admin' check — a match is found, so login succeeds. Root cause: concatenation means the database cannot separate the developer's original code structure from the attacker's input.
Q(b) 2 pts Tester types ' OR 1=1# as username. What does WHERE become? What is this called? ›
Model Answer
The WHERE clause becomes: WHERE name='' OR 1=1 Because 1=1 is always true, this is called a tautology attack. Every row passes the condition so the entire table is dumped and returned.
Q(c) 2 pts Developer says "I validate input with JavaScript on the frontend — that's enough." Is this reliable? ›
Model Answer
No. An attacker can easily use curl, Postman, or a web intercepting proxy to modify the HTTP request before it reaches the server, bypassing frontend validation entirely. Input validation must be performed server-side. Even then, the only truly reliable fix is using parameterized queries — not validation alone.
Q(d) 3 pts The developer rewrites using parameterized queries. Explain exactly why this prevents injection. ›
Model Answer
The query structure is sent first to the database with placeholders: WHERE name=? AND password=? The database compiles the query and locks in the structural logic. User data values are sent separately afterwards and treated as pure data literals — never as executable code. Even if the attacker injects admin'#, the database reads it as a literal string value — not SQL logic. Injection is completely prevented because code and data are structurally separated.
Question Set 2

UPDATE/Semicolon Injection Scenario

Scenario

An update query is built as:

$sql = "UPDATE credential SET salary='$input_salary' WHERE name='$input_name'";

The tester injects into $input_name:

Alice'; UPDATE credential SET salary=999999 WHERE name='Alice';#
Q(a) 3 pts What do the semicolons do? What do the resulting queries execute? ›
Model Answer
The first single quote ' closes the string literal. The first semicolon ; ends the execution of the original statement. The second statement immediately begins — a new UPDATE statement. The hash # erases the rest of the developer's trailing SQL code. UPDATE credential SET salary='Alice'; UPDATE credential SET salary=999999 WHERE name='Alice'; The first statement fails silently. The second statement permanently modifies Alice's salary to 999,999.
Q(b) 2 pts Why is this attack particularly hard to detect? ›
Model Answer
The web application behaves completely normally — there is no visible database crash, error output, or browser interruption. The unauthorized data modification happens silently in the background with no visible indications to either the victim or the system.
Q(c) 2 pts The database account is read-only. How does this limit the damage? What principle is this? ›
Model Answer
A read-only account is blocked from running modification commands like UPDATE, DELETE, or DROP — the injected second statement is immediately rejected by the database. The attacker can read data but cannot modify or destroy it. This demonstrates the principle of Least Privilege — limiting DB account permissions minimizes the blast radius of a successful injection.
Q(d) 3 pts Explain the three-layer architecture and how SQL injection exploits the middle layer. ›
Model Answer
The architecture consists of: Browser → Web Server → Database. Users have no direct network access path to the database. When the web server concatenates user input directly into SQL strings, the attacker controls what the server transmits. The attacker is effectively using the web server as a proxy to reach and talk to the backend database. Parameterized queries break this execution pathway by ensuring user input can never be compiled into executable SQL code.
