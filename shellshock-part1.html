Shellshock: Part 1

Lecture 14 — CSC351 Application Security

Concept

The Story Starts Here

It's 2014. A French cybersecurity researcher is poking around in the source code of a piece of software that almost every Linux computer in the world uses every single day.

He finds something strange.

A bug. A really bad one.

And then he checks the git history — the record of every change ever made to this software — and he finds that this bug has been sitting there, quietly, since 1989.

Twenty-five years. Hidden in plain sight. In software running on millions of servers all over the world.

That's Shellshock. And to understand why it's so dangerous, you first need to understand something much simpler — what a shell is, and how it talks to itself.

Concept

What Is a Shell?

You know how on Windows you can open a black box called Command Prompt or PowerShell and type commands?

That black box is a shell. It's the thing that sits between you — the human — and the actual brain of the operating system called the kernel.

The kernel is like the engine of a car. You can't just reach in and touch the engine with your bare hands. The shell is the steering wheel and pedals — the interface that lets you control the engine without touching it directly.

Different operating systems have different shells:

  • Windows has PowerShell.
  • macOS used to use Bash, then after Shellshock it switched to ZSH (which Ajan mentioned is the C shell family).
  • Linux still uses Bash as its standard shell today.

Bash stands for **Born Again Shell**. Ajan wrote it on the slides. It's a bit of a nerdy joke — it's a rewrite of the original Unix shell (called sh), so it's "born again."

And Bash is exactly where the Shellshock bug lives.

Concept

Functions in Bash — The Innocent Part

Before you can understand the bug, you need to understand one thing Bash can do: define functions.

A function in Bash is just like a function in any programming language. You give it a name, you say what it should do, and later you can call it by that name.

In Bash, you define a function like this:

foo() { echo "hello world"; }

Now whenever you type foo in the terminal, Bash runs echo "hello world" and you see hello world on screen.

Totally harmless. This is just normal Bash functionality.

Processes

Parent Processes and Child Processes — The Stage Is Set

Here's a concept Ajan spent real time on in the lecture.

When you open a program on your computer, that program lives in a slice of your RAM. It's a process — a running program with its own little world in memory.

Now sometimes a process opens another process from inside itself. The new one is called the child process. The original is the parent process.

Think of it like a manager at work. The manager has their own desk, their own files, their own knowledge. When the manager sends an employee off to do a job, that employee is the child process. They have their own workspace. They start fresh.

Here's the critical rule: functions and variables defined in the parent process do NOT automatically pass to the child process. If the manager knows a secret password, the new employee they send out doesn't automatically know it. The manager has to deliberately export it to them first.

In Bash, that export command is literally called export.

Variables

Environment Variables — The Messenger

So how does a parent process send information to a child process?

Through something called environment variables.

These are like little sticky notes that the parent pastes onto the child when it's created. Name=Value pairs. The child inherits all of them automatically.

You use these all the time without knowing it. When your web browser sends an HTTP request, the web server reads headers like User-Agent and Cookie — and when it creates a child process to run a script, it passes those headers as environment variables to the child.

That detail — HTTP headers becoming environment variables — is the entire bridge that makes Shellshock remotely exploitable. But we'll get to that in Part 2.
Vulnerability

The Vulnerability — Where Bash Goes Too Far

Here's the bug. The actual thing Bash does wrong.

When a child Bash process starts up, it looks at all the environment variables it inherited from the parent. For each one, it checks: "Is this a function definition?"

It recognizes a function definition by looking for a specific pattern at the start of the value — something that looks like () {.

If it sees that pattern, it thinks: "Oh! My parent exported a function to me. Let me convert this back into a real function."

So it runs a piece of internal code called parse_and_execute on that value.

And here's the bug. parse_and_execute doesn't stop at the end of the function.

Imagine you receive a note from your manager that says: "Meeting at 9am. Fire the entire accounting department."

A sensible person reads the first sentence and stops — that's the instruction they were given. But Bash reads the entire note and does everything it says.

The function definition ends at the }. But Bash keeps reading. And anything written after the closing } gets executed as a real shell command.

This is the Shellshock vulnerability. One extra semicolon and a command after the function body — and Bash runs it.

POC

The Proof of Concept Test

Ajan showed the exact command that proves whether a system is vulnerable. You run this in a terminal:

env x='() { :; }; echo vulnerable' bash -c "echo test"

Let's break down what this does, piece by piece, so it actually makes sense:

  • env — sets an environment variable for just this one command.
  • x='() { :; }; echo vulnerable' — sets a variable called x. Its value looks like a function definition (() { :; } — a function that does nothing, the : is Bash's no-op). But after the closing } there's a semicolon and then echo vulnerable.
  • bash -c "echo test" — starts a new Bash process and tells it to run echo test.

If the system is vulnerable: the new Bash process starts, imports the environment variable x, sees the () { pattern, decides it's a function, and runs parse_and_execute on it. parse_and_execute doesn't stop at }. It runs echo vulnerable. Then it runs echo test.

Output: **vulnerable** then **test**.

If the system is patched: Bash imports the variable, sees the function, stops reading at }, and ignores the echo vulnerable. It only runs echo test.

Output: **test**.

That's the entire test. You can tell in one second whether a machine is vulnerable.

High Risk

Why Is This Dangerous Beyond a Local Terminal?

Okay, so you can inject a command into a new Bash process. So what? An attacker still needs to be logged in to run the env command, right?

Wrong. And this is where it gets scary.

Remember how environment variables are passed from parent to child? And remember how HTTP request headers become environment variables when a web server handles a request?

Here's the attack chain:

Step 1
The web server receives an HTTP request.
Step 2
The web server creates a child process to run a CGI script.
CGI stands for Common Gateway Interface — an old-school way of generating dynamic web content using shell scripts. Still used on many servers.
Step 3
The web server passes all the HTTP request headers as environment variables.
User-Agent, Referer, Cookie, etc. become environment variables inside that child process.
Step 4
The child process runs Bash, parses environment variables, and executes the payload.
The attacker just executed code on the server. From a browser. Without any login. Without any credentials. All they had to do was put the attack payload in an HTTP header.
Quick Revision

Quick Revision

  • Bash = Born Again Shell, the standard shell in Linux, sits between the user and the OS kernel
  • Shell functions can be passed from parent process to child process via environment variables
  • Shellshock = CVE-2014-6271, a bug in Bash that existed since 1989 and was discovered in 2014
  • The bug is in parse_and_execute — when Bash converts an environment variable back into a function, it doesn't stop reading at the closing } and executes everything after it
  • Proof of concept test: env x='() { :; }; echo vulnerable' bash -c "echo test" — outputs "vulnerable" if unpatched, just "test" if patched
  • Remote exploitation requires two conditions: vulnerable unpatched Bash AND a CGI script on the web server
  • The attack chain: attacker sends HTTP request → Apache creates CGI child process → Apache passes headers as environment variables → vulnerable Bash executes the payload hidden in the header
  • curl flags: -A sets User-Agent, -e sets Referer, -H sets custom header — all become environment variables in the CGI child process
  • Without CGI, no Bash child process is created from HTTP requests, so the attack has no path in
