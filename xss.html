Cross-Site Scripting

Lecture 11 — CSC351 Application Security

Concept

The Problem That Started Everything

Okay so imagine you're running a social network. Users can write bios, leave comments, post stuff. You store whatever they type in your database, then show it to other people.

Simple enough, right?

Now imagine someone types this into their bio field: <script>alert('hacked')</script>

You save it. Someone else visits their profile. Your server sends back the page. The browser reads it — and sees a <script> tag sitting right there in the HTML.

The browser has no idea that's evil. It came from your server. So it runs it.

That's the whole problem. Your website accidentally became the delivery truck for someone else's code.

Concept

Why Does the Browser Trust It?

Think about a letter with a forged return address.

If you get a letter that looks like it's from your bank, you open it without checking twice. The browser does the same thing. It doesn't care who wrote the code — it only checks where it came from.

If the code arrives from example.com, the browser treats it as example.com's code. Full stop.

So when an attacker gets their JavaScript into your page, the browser gives it the exact same trust it gives your own scripts. That's the core mechanic. That's why XSS is powerful.

Concept

Type 1 — Reflected XSS

Here's the story.

You visit example.com/search?q=shoes. The page shows: "You searched for: shoes."

Now notice what's happening — whatever you put in that URL, the server reflects it back onto the page. That's called reflective behavior.

An attacker realizes: what if I put JavaScript there instead of "shoes"?

They craft a URL like example.com/search?q=<script>stealCookies()</script> and they trick the victim into clicking it. Maybe they hide it inside a QR code, a phishing email, a disguised link.

The victim clicks. The browser sends the request. The server reflects the script back in the HTML. The browser runs it.

The attack happens once, in real time. It's not saved anywhere. That's why it's called reflected — it bounces off the server back to the victim.

Concept

Type 2 — Stored (Persistent) XSS

Same idea. But this time, the attacker puts JavaScript into a field that gets saved to the database. A profile bio. A comment. A support ticket.

Now the malicious code just sits there, patient.

Every single user who loads that page later gets the script delivered by your server automatically. No crafted URL. No trick click needed. Just visiting the page is enough to get hit.

Ajan used Facebook profiles as the example. You write JavaScript into your profile, other people visit your profile, they get attacked. Your profile page becomes a channel from the attacker to every victim.

This is why stored XSS is considered more dangerous — it scales. One payload, unlimited victims. Alice and Bob textbook: it's "an attack the user is unable to protect themselves against." Because it comes from a domain they already trust.
Impact

What Can XSS Actually Do?

Three things, and all three make sense once you remember that JavaScript inside a browser has power over everything on that page.

01 — Deface
Web defacing
JavaScript can reach into the page's DOM and change any text, any image, any element. It can make a legitimate bank site show fake balances, fake buttons, anything.
02 — Forge
Spoofing requests
JavaScript can fire off HTTP requests using fetch() or XMLHttpRequest. The injected code can send requests to the server pretending to be the user — same as how web apps normally work, just weaponized.
03 — Steal
Stealing information
JavaScript can read session cookies (if they're not protected), read any personal data displayed on the page, and even read things stored locally in the browser. The attacker receives all of this on their own server.
Defenses

The Three Defenses

Think of three checkpoints in an assembly line.

Layer 1
Input Validation (before saving)
When the user submits data, check it before you store it. If a name field contains <script>, reject it. Don't let evil through the front door.
Layer 2
Output Encoding (before displaying)
When you pull data out of the database and put it on screen, convert dangerous characters into harmless text first. <script> becomes &lt;script&gt;. The browser renders it as visible text, not runnable code. It's ugly. It's also harmless.
Layer 3
CSP Header (last line of defense)
Content Security Policy is an HTTP response header that tells the browser: "Only run scripts from these approved sources. Block everything else." Even if someone bypasses the first two checkpoints, CSP can stop the injected script from running.
Alice and Bob textbook calls applying all three a "defense in depth" strategy. Ajan mentioned all three in class. Know all three.
⚠ Exam trap: Output encoding must be applied when displaying data, not when storing. Storing encoded data is a mistake because you lose the original data.
⚠ High-Stakes Distinction

The Biggest Trap — XSS vs CSRF

Ajan called this out directly in the Block 2 exam review. Students lost points for mixing these up. Don't be that person.

XSS
Attacker gets their own code to run in the victim's browser
  • The weapon is the attacker's JavaScript
  • Attacker's script executes on the trusted page
  • Can actively read and steal cookies, DOM data, tokens
  • The attacker's code crosses through the target site
CSRF
Attacker tricks the victim's browser into sending a legitimate request
  • The weapon is the victim's own session cookie
  • No attacker code runs in the browser at all
  • Uses cookies without ever reading them
  • The request crosses from one site to another
Ajan deducts 2 points for describing cookie theft in a CSRF answer, or describing forged requests in an XSS answer. Know which is which.
Exam Tips

Exam / Scenario Tips

Ajan loves giving you a scenario where a website shows user-submitted content and asking you to trace the exact attack path.

The trap is usually in the type. If he says "a user updates their profile bio and other users are affected," that's stored XSS — don't say reflected. If he says "a search bar echoes your input back," that's reflected XSS — don't say stored.

The other trap is the countermeasure question. If he asks where to apply output encoding, the answer is: at the point where data is displayed on the page — not when it's stored. Storing encoded data is a mistake because you lose the original data.

And if he asks why XSS is different from all other injection attacks — the answer is: XSS executes on the client side (browser), every other injection executes on the server side. That distinction earns marks.

Quick Revision

Quick Revision

  • XSS happens because the browser trusts all code that comes from a domain, regardless of who actually wrote it
  • Reflected XSS: payload travels in the URL, server echoes it back, not stored, victim must be tricked into clicking the crafted link
  • Stored XSS: payload saved to database, delivered to every user who loads the page, no click required from victims
  • Three damages: web defacing (DOM manipulation), spoofing requests (JS can send HTTP), stealing info (cookies, page data, local storage)
  • Three defenses in order: input validation before saving → output encoding before displaying → CSP header as last resort
  • XSS ≠ CSRF: XSS runs attacker's code in the browser, CSRF uses the victim's own session to send a forged request — mixing these up is a confirmed deduction
