Lecture 13 · Part 2 · Exam Prep

Clickjacking — Exam Practice Questions

Scenario-based questions with model answers. Click any question to reveal the answer.

Quick Notes

  • Golden rule — Ajan said this word for word:
    • "The top layer is always clickable" — the user always clicks the topmost iframe regardless of what they physically see on the screen
  • Two overlapping iframes:
    • Bottom: real target page, visible (decoy target background)
    • Top: transparent (opacity: 0), completely invisible but intercepts and receives all clicks
  • Coordinate system:
    • Origin = top-left corner = (0,0)
    • top = pixels down, left = pixels right, width + height = iframe dimensions
    • Attacker dynamically adjusts top/left margins until the invisible target button sits exactly over the real visible decoy button
  • Two stacking rules:
    • z-index → larger integer values sit on top
    • No z-index → the element declared last in the HTML markup sits on top
  • Attack variants:
    • Likejacking → invisible Facebook/decal Like button over decoy prize links
    • Fake login → transparent input form fields capture target credentials
    • Small iframe overlay → tiny iframe covering one specific button only, leaving the rest of the page working normally so the victim suspects nothing
    • Sequence of clicks → maps multiple consecutive clicks across steps — hardest variant to execute
  • Three defense generations:
    • Gen 1 — Frame buster/killer: JS checks if window.top === window.self and redirects parent if false. Dead. Killed by: attacker setting sandbox="allow-forms" on the parent iframe, which disables Javascript execution inside, so the buster logic never runs.
    • Gen 2 — X-Frame-Options (DENY / SAMEORIGIN): Deprecated. Ajan said this explicitly in class — still works on legacy browsers only.
    • Gen 3 — Content-Security-Policy: frame-ancestors: 'none' or 'self' or specific URLs. Current standard. Configured at Apache/Nginx web server header levels, applying to all pages automatically.
  • SOP inside iframes:
    • Same origin → parent context can read/write child iframe DOM tree
    • Different origin → Same-Origin Policy blocks DOM tree reads and writes
    • This is why the attacker makes the top target iframe fully transparent — they cannot read the victim's cross-origin page data anyway, so they only intercept physical inputs.
The exam traps:
Saying X-Frame-Options is the current standard → deduction.
Saying a frame buster still works against attackers → deduction.
Saying CSRF tokens protect against clickjacking → wrong (the victim is interacting with their own real, authenticated session, so the tokens are completely valid).
Question Set 1

SEED Lab Clickjacking Setup

Scenario

Ajan demonstrated this exact attack code in the SEED lab. An attacker hosts this page on attacker32.com. The victim is logged into bank32.com.

<!-- Attacker's page on attacker32.com --> <iframe id="top" src="http://attacker32.com/fake_prize.html" style="position:absolute; top:0; left:0; width:500px; height:400px; opacity:1.0;"> </iframe> <iframe id="bottom" src="http://bank32.com/transfer?to=attacker&amount=5000" style="position:absolute; top:0; left:0; width:500px; height:400px; opacity:0.0;"> </iframe>
Q(a) 2 pts Which iframe does the victim see? Which iframe actually receives the click? Explain using Ajan's golden rule. ›
Model Answer
The victim physically sees fake_prize.html because its opacity is set to 1.0. The bottom bank32.com transfer page is completely invisible (opacity 0.0). However, because there are no z-index declarations, the element declared last in the HTML markup (in this case, #bottom) sits on top. Thus, the invisible #bottom iframe intercepts and receives the victim's physical click. **Ajan's golden rule:** "The top layer is always clickable", meaning the browser processes clicks on the topmost layer in the stacking order, regardless of opacity.
Q(b) 2 pts The victim never visited bank32.com directly. Why does the bank still process the transfer as legitimate? ›
Model Answer
The victim is already logged into bank32.com in their browser. When the browser loads the bank page inside the iframe, it automatically attaches the victim's session cookies to all outgoing requests. The bank server processes the request as a fully valid, authenticated transfer transaction, indistinguishable from an intentional action.
Q(c) 3 pts Name all three generations of Clickjacking defense. State dead/deprecated/current for each. For the dead one, explain exactly how attackers defeated it. ›
Model Answer
Gen 1 — Frame buster script (Dead): JavaScript blocks checking if the current window is top. Attacker defeats this by configuring the iframe with sandbox="allow-forms", which disables script execution within the frame so the buster code never runs. Gen 2 — X-Frame-Options (Deprecated): Header responses (DENY, SAMEORIGIN). Works on older legacy browsers only, and is deprecated. Gen 3 — CSP frame-ancestors (Current Standard): Header responses (frame-ancestors 'none' / 'self'). Configured at the web server level to enforce site-wide security.
Q(d) 3 pts Explain why CSRF tokens do NOT protect against Clickjacking, and why SOP does NOT protect against it either. ›
Model Answer
CSRF Tokens: The real target page is loaded legitimately inside the iframe, meaning the form contains a completely genuine, valid CSRF token generated for the user's active session. Since no request parameters are forged, CSRF defenses have nothing to detect. SOP (Same-Origin Policy): SOP blocks scripts on one origin from reading/writing data inside document nodes of another origin. Clickjacking does not read or write data inside the target frame — it simply lets a physical user click pass through to trigger a button. SOP has nothing to restrict.
Question Set 2

Header Bypass & Attack Variants

Scenario

A security researcher captures this HTTP response header from bank32.com:

HTTP/1.1 200 OK Content-Type: text/html X-Frame-Options: SAMEORIGIN

An attacker successfully embeds bank32.com inside an iframe on evil.bank32.com and launches a Clickjacking attack.

Q(a) 2 pts How is the attacker able to bypass X-Frame-Options: SAMEORIGIN in this scenario? ›
Model Answer
The SAMEORIGIN directive allows a page to be framed by any page sharing the exact same parent origin. Since the attacker hosts their page on evil.bank32.com, depending on how subdomains are evaluated by older browser implementations, it may be treated as sharing the same top-level origin, allowing the subdomain to successfully frame the site.
Q(b) 2 pts What should the developer have used instead? Write the exact header and explain one advantage it has over X-Frame-Options. ›
Model Answer
Recommended Header: Content-Security-Policy: frame-ancestors 'self' Advantage: frame-ancestors allows the developer to supply a whitelist of multiple safe URLs that can frame the page, whereas X-Frame-Options can only accept one single origin. Furthermore, CSP represents the current web security standard, while X-Frame-Options is deprecated.
Q(c) 3 pts The attacker's page uses a small iframe overlay variant instead of a full-page iframe. Explain how this variant works and why it is harder for the victim to detect. ›
Model Answer
Rather than framing the entire bank page over the attacker's decoy, the attacker targets a single critical action and overlays a tiny, transparent iframe (sized identically to one button) exactly over a visible decoy button. The rest of the page remains the attacker's normal, legitimate-looking decoy site. If the victim clicks anywhere else on the page, the decoy reacts correctly. The victim does not suspect any security issue because most of the UI responds exactly as expected.
Q(d) 3 pts Ajan demonstrated SOP behavior inside iframes using three iframes — A and B from attacker32.com, C from bank32.com. Explain which iframe the parent can and cannot read, and why this matters for Clickjacking. ›
Model Answer
The parent page can read and write inside iframes A and B because they share the same origin (attacker32.com) and SOP allows DOM access. The parent cannot access the DOM of iframe C because it belongs to a different origin (bank32.com) and SOP blocks cross-origin access. This matters because Clickjacking is explicitly designed to bypass SOP limitations. Since the attacker's script cannot extract the bank token, transfer form fields, or user account balances from iframe C, their only option is to lure the authenticated user into physically clicking the invisible transfer button itself.
