SQL Injection
Lecture 12 — CSC351 Application Security
The Story

The Story Starts Here

Okay so imagine your school has a secret candy storeroom.

Only teachers can go in. There's a guard at the door. You walk up and the guard asks: "What's your name and your teacher code?"

You say "Mr. Smith, code 1234" — the guard checks his list and lets you in.

Simple enough, right?

The Attack

What If You Were Really Clever?

What if instead of your name, you said:

"Mr. Smith OR everyone is a teacher"

The guard isn't very smart. He reads the whole thing out loud from his list: "Is this person Mr. Smith — OR — is everyone a teacher?"

Well... everyone IS a teacher in that second part. That's always true.

So the guard goes "yep, that checks out" and opens the door.

You just got into the candy storeroom without knowing the real teacher code.

That's SQL Injection. You didn't break the lock. You didn't climb through a window. You just said something sneaky to the guard and he let you in because he was too dumb to know the difference.

The Mechanic

So What's the "Guard" in Real Life?

In a real website, the guard is your database.

When you type your username and password into a login page, the website takes what you typed and builds a question to ask the database. Like this:

"Hey database, do you have a user named 'alice' with password 'abc123'?"

The database checks its list. If it finds a match, it says yes and you're in.

The problem is how the website builds that question.

Root Cause

The Dangerous Way to Build the Question

Some websites do it like this — they just stick your input directly into the question like gluing pieces of paper together:

"Find me a user where name = " + whatever you typed + " and password = " + whatever you typed

So if you type a normal name like alice, the question is totally fine.

But what if you type something sneaky instead?

The Injection

The Sneaky Input

You type this as your username:

' OR 1=1 --

The website glues it in like always. Now the question becomes:

"Find me a user where name = '' OR 1=1 -- and password = ..."

Three things just happened.

The OR 1=1 part means "OR this is always true." One always equals one. So the whole condition becomes true for every single person in the database.

The -- part is a comment. It tells the database "ignore everything after this." So the password check? Gone. Erased.

The database reads it and goes "yep, that's true for everyone" and hands over everything.

You just walked into the candy storeroom.

Escalation

It Gets Worse

The sneaky input doesn't just let you peek in. You can add a second command after a semicolon.

Imagine typing:

alice; DROP TABLE users;

Now the database gets two instructions:

First: find the user alice.

Second: DELETE THE ENTIRE USER TABLE.

Every account. Every password. Every order. Every profile. Gone forever.

The Hoffman textbook calls this the nuclear option. One input field. Entire database wiped.

Why It Works

Why Does the Database Fall for It?

Because nobody told it to be suspicious.

When the website sends the database a question, it sends it as one big string of text. The database just reads it and does what it says.

It can't tell the difference between:

The part the developer wrote — the safe instructions.

The part the attacker added — the sneaky commands.

It all looks the same to the database. One big instruction. Execute it all.

Comment Trick

The Comment Trick — The Attacker's Eraser

Ajan taught three ways to write comments in MySQL. Memorize these — they come up in exams.

#

ignore everything after this

--

ignore everything after this

/* */

ignore everything between these

Why does an attacker care about comments?

Because sometimes the developer writes extra safety checks at the end of the query. Password verification, for example. The attacker uses a comment to erase those checks. Like using whiteout on the part of the sentence they don't want read.

Tautology

The WHERE 1=1 Trick — The Magic Pass

Ajan spent a lot of time on this in the demo.

WHERE is the part of a database question that filters results. Like "only show me students WHERE their grade is above 80."

If the condition is always true — like WHERE 1=1 — every single row passes the filter. One always equals one. Every student gets shown. The whole table dumps out.

Ajan literally showed this live in the terminal. He typed SELECT * FROM credential WHERE 1=1 and every employee record appeared on screen.

This trick has a fancy name: a tautology attack. A tautology is just a statement that's always true no matter what. Smart-sounding word, simple idea.
Damage Levels

Three Damage Levels

Read

SELECT — dump the entire table, read all records, export everything

Modify

UPDATE — silently change data, alter salaries, change passwords

Destroy

DROP TABLE — delete everything, nuclear option, irreversible

The Fix

The Fix — Two Steps Instead of One

Okay so here's how smart developers stop this.

Instead of building one big string and sending it to the database, they do it in two trips.

1
Send the question structure, with empty blanks where the data will go.
"Find me a user where name = ? and password = ?"
The database reads this, understands the structure, and locks it in. The intention is set in stone.
2
Send the actual values to fill in the blanks.
username = alice, password = abc123
Now here's the magic part.

Even if someone types ' OR 1=1 -- as their username, the database treats it as data. It looks for a user whose name is literally the string ' OR 1=1 --.

Nobody has that name. Login fails. Attack blocked.

The attacker's sneaky code never gets executed. It's treated as a shopping list item, not as an instruction.

This is called a parameterized query. The Alice and Bob textbook is very clear: "Always use parameterized queries." It's the only fix that actually works at the root.
Backup Safety

The Last Line of Defense — Least Privilege

Even with parameterized queries, smart developers add one more thing.

The database account that your website uses to talk to the database should only have the minimum permissions it needs.

If your page only needs to read data, use a read-only account. That account literally cannot run DELETE or DROP or UPDATE. It doesn't have permission.

So even if an attacker somehow breaks through, they can look at data but they can't destroy it. The damage is limited.

Ajan calls this Least Privilege — you know this from Block 1. Give everyone — including software — only the access they actually need and nothing more.
Exam Tips

Exam / Scenario Tip

Ajan will show you a piece of code where username is glued directly into a SQL string with a + sign. He'll ask you to trace the attack.

The trap is saying "the input wasn't validated." That gets partial marks. What gets full marks is explaining the two-layer problem: the database receives one combined string, it can't separate the developer's code from the attacker's injection, so it executes both.

The defense answer is not "validate the input." It's parameterized queries. Validation alone can be bypassed. Parameterized queries fix the root cause by making data structurally impossible to interpret as code.

If he asks about WHERE 1=1 specifically — call it a tautology attack and explain that a tautology is a condition that's always true, so every row in the table passes the filter.

Quick Revision

SQL Injection = tricking the database by typing sneaky code into an input field that gets glued directly into a database query
String concatenation is the root cause — the developer pastes user input directly into the SQL command, so the database can't tell instructions from injected code
WHERE 1=1 = tautology attack — always true, dumps every row in the table
# and -- and /* */ = SQL comment characters — attackers use these to erase the legitimate parts of the query after their injection point
Three damage levels: read everything (SELECT), silently modify data (UPDATE), destroy everything (DROP TABLE)
Parameterized queries = the real fix — send query structure first, data second, so data can never be interpreted as code
Least privilege = the backup safety net — if the database account is read-only, a successful attack can read data but cannot destroy it
SQL injection executes server-side in the database — XSS executes client-side in the browser — don't mix these up
